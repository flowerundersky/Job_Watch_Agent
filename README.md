# Job Watch Agent

本项目是一个本地运行的招聘监控智能体，当前实现的是“第一版代码”。
它的目标不是一次性覆盖所有招聘站，而是先跑通一条稳定的最小闭环：
读取配置 -> 抓取页面 -> 清洗和去重 -> 对比历史数据 -> 做岗位匹配 -> 生成本地 PDF 报告。

## 第一版代码

这一版代码已经具备以下能力：

1. 读取本地 `config.yaml` 配置。
2. 访问配置中的招聘页面 URL 列表。
3. 使用 `requests + BeautifulSoup` 抓取并解析页面中的岗位信息。
4. 对岗位信息做基础清洗和去重。
5. 将本次抓取结果与上一次保存的历史快照进行比对。
6. 使用规则型匹配器判断岗位是否匹配关键词或公司名。
7. 生成本地 PDF 报告。
8. 将本次抓取结果保存为历史快照，供下次运行比较。
9. 支持可切换的模型后端：本地规则匹配或 OpenAI 兼容中转站 API。

### 第一版代码能做什么

#### 1. 自动跑通单次监控流程

运行 `main.py` 后，程序会按照配置中的 `sources` 逐个访问页面，并完成整轮处理。
它不是后台常驻服务，而是“执行一次、输出一次结果”的批处理模式。

#### 2. 抓取静态页面中的岗位卡片

当前抓取器主要面向静态 HTML 页面，能从常见的岗位卡片、列表项、`article`、`li`、`a` 标签中提取信息。
如果页面里本身已经包含岗位标题、公司名、地点、发布时间和链接，第一版通常可以识别出来。

#### 3. 做基础岗位标准化

代码会对文本做基础空白归一化，并将岗位对象序列化为统一结构。
这有助于后续做去重、比对和落盘。

#### 4. 做新旧数据比对

程序会读取 `data/history.json` 作为历史快照，和本次抓取结果进行对比。
它可以识别三类变化：

- 新增岗位
- 内容变化岗位
- 已消失岗位

#### 5. 做基础岗位匹配，或切换到 OpenAI 兼容后端

默认情况下，第一版使用规则型匹配器。
如果你在 `config.yaml` 里把 `model_backend.backend` 改成 `openai_compatible`，并填好中转站的 `api_base_url` 和 `api_key`，程序就会把岗位信息发给 OpenAI 兼容接口进行分析。
模型需要返回 JSON 结果，程序会解析 `matched`、`score` 和 `reasons` 字段，用于后续报告展示。

这一步的作用是先把业务流程接起来，同时保留真实模型后端的接入方式。

#### 6. 生成 PDF 报告

程序会把本次运行摘要、匹配岗位和变化岗位写入本地 PDF 文件，默认输出到 `output/` 目录。
这份报告适合人工快速浏览，查看当天有哪些新岗位和哪些岗位更值得关注。

### 第一版代码的业务流程

一次完整运行的顺序如下：

1. 读取 `config.yaml`。
2. 创建必要的本地目录。
3. 从 `sources` 中逐个抓取页面。
4. 提取岗位标题、公司、地点、链接、发布时间等字段。
5. 对岗位进行清洗和去重。
6. 读取历史快照并比较差异。
7. 对新增岗位和变化岗位做匹配判断。
8. 生成 PDF 报告。
9. 保存新的历史快照。

### 第一版代码的输入

当前版本主要依赖这些输入：

- `config.yaml` 中的 `sources`
- `config.yaml` 中的 `keywords`
- `config.yaml` 中的 `companies`
- `config.yaml` 中的 `runtime` 配置
- `config.yaml` 中的 `model_backend` 配置

配置样例：

```yaml
sources:
  - https://example.com/careers
keywords:
  - Python
  - 数据分析
companies:
  - 示例科技
runtime:
  output_dir: output
  data_dir: data
  logs_dir: logs
  timeout_seconds: 10
model_backend:
  backend: rule
  api_base_url: ""
  api_key: ""
  model: gpt-4o-mini
  timeout_seconds: 30
  temperature: 0.2
  max_tokens: 512
```

### 第一版代码的输出

每次运行后，程序会生成这些结果：

- 终端输出运行摘要
- `output/` 下的 PDF 报告
- `data/history.json` 历史快照文件

## 第一版代码不能做什么

为了避免误解，下面这些能力当前不支持，或者只保留了接口，没有做完整实现：

### 1. 不能自动搜索全网招聘信息

当前版本只会抓取配置文件里明确写入的 URL。
它不会自动使用搜索引擎去发现新的招聘页面，也不会自动遍历全网公司站点。

### 2. 不能稳定处理复杂的动态网页

当前抓取逻辑使用 `requests` 请求页面 HTML，再用 `BeautifulSoup` 解析。
如果页面内容依赖 JavaScript 渲染、滚动加载、异步接口或登录态，第一版通常无法直接处理。

### 3. 不能绕过反爬和验证码

当前版本没有做代理池、验证码识别、浏览器自动化、指纹伪装等反爬处理。
如果目标站点限制请求，它可能会抓取失败。

### 4. 不能保证适配所有招聘网站结构

第一版是通用解析策略，不是每个站点的深度适配器。
如果页面结构特殊，可能需要后续补专门的站点解析器。

### 5. 不能做真正的 LLM 推理匹配

当前的 `LLMAnalyzer` 只是一个规则型代理，目的是先把业务流程跑通。
它会按关键词和公司名打分，但还没有接入真实的大模型 API。

### 6. 不能长期后台守护或自动定时运行

当前版本是一次性执行模型，不包含调度器、守护进程或定时任务框架。
如果要每天自动运行，需要后续再接 cron、系统任务或独立调度模块。

### 7. 不能做复杂的职位画像建模

第一版只支持简单的关键词和公司名匹配。
它不会分析完整的候选人画像、技能图谱或多层级岗位偏好。

## 目录说明

- `main.py`：程序入口，负责读取配置并启动流程。
- `config.yaml`：运行配置，包含来源站点、关键词、公司名和目录设置。
- `src/scraper.py`：页面抓取与 HTML 解析。
- `src/cleaner.py`：岗位清洗和去重。
- `src/comparator.py`：新旧快照比对。
- `src/analyzer.py`：岗位匹配逻辑，当前为规则型代理。
- `src/reporter.py`：PDF 报告生成。
- `src/storage.py`：历史快照保存与读取。
- `src/pipeline.py`：把各模块串成完整业务流程。
- `tests/`：pytest 测试集。

## 运行方式

安装依赖后，直接运行：

```bash
python main.py
```

也可以指定其他配置文件：

```bash
python main.py /path/to/config.yaml
```

## 测试方式

运行测试：

```bash
pytest
```

当前测试覆盖了：

- 配置读取
- 岗位清洗与去重
- 新旧比对
- 规则匹配器
- PDF 报告生成
- 历史快照保存与读取
- HTML 岗位提取

## 后续版本计划

如果后面继续迭代，这一版最自然的升级方向是：

1. 增加更多站点适配器。
2. 引入浏览器自动化，支持动态页面。
3. 接入真正的 LLM 分析接口。
4. 增加日志、重试、失败隔离和调度。
5. 细化报告内容和岗位筛选策略。
