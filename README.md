# Job Watch Agent

本项目是一个本地运行的招聘监控智能体，当前实现的是“第一版代码”。
它的目标不是一次性覆盖所有招聘站，而是先跑通一条稳定的最小闭环：
读取配置 -> 抓取页面 -> 清洗和去重 -> 对比历史数据 -> 做岗位匹配 -> 生成本地 PDF 报告。

## 第一版代码

这一版代码已经具备以下能力：

1. 读取本地 `config.yaml` 配置。
2. 访问配置中的招聘页面 URL 列表。
3. 使用 `requests + BeautifulSoup` 抓取并解析页面中的岗位信息。
4. 对岗位信息做基础清洗和去重。
5. 将本次抓取结果与上一次保存的历史快照进行比对。
6. 使用规则型匹配器判断岗位是否匹配关键词或公司名。
7. 生成本地 PDF 报告。
8. 将本次抓取结果保存为历史快照，供下次运行比较。
9. 支持可切换的模型后端：本地规则匹配或 OpenAI 兼容中转站 API。

### 第一版代码能做什么

#### 1. 自动跑通单次监控流程

运行 `main.py` 后，程序会按照配置中的 `sources` 逐个访问页面，并完成整轮处理。
它不是后台常驻服务，而是“执行一次、输出一次结果”的批处理模式。

#### 2. 抓取静态页面中的岗位卡片

当前抓取器主要面向静态 HTML 页面，能从常见的岗位卡片、列表项、`article`、`li`、`a` 标签中提取信息。
如果页面里本身已经包含岗位标题、公司名、地点、发布时间和链接，第一版通常可以识别出来。

#### 3. 做基础岗位标准化

代码会对文本做基础空白归一化，并将岗位对象序列化为统一结构。
这有助于后续做去重、比对和落盘。

#### 4. 做新旧数据比对

程序会读取 `data/history.json` 作为历史快照，和本次抓取结果进行对比。
它可以识别三类变化：

- 新增岗位
- 内容变化岗位
- 已消失岗位

#### 5. 做基础岗位匹配，或切换到 OpenAI 兼容后端

默认情况下，第一版使用规则型匹配器。
如果你在 `config.yaml` 里把 `model_backend.backend` 改成 `openai_compatible`，并填好中转站的 `api_base_url` 和 `api_key`，程序就会把岗位信息发给 OpenAI 兼容接口进行分析。
模型需要返回 JSON 结果，程序会解析 `matched`、`score` 和 `reasons` 字段，用于后续报告展示。

这一步的作用是先把业务流程接起来，同时保留真实模型后端的接入方式。

#### 6. 生成 PDF 报告

程序会把本次运行摘要、匹配岗位和变化岗位写入本地 PDF 文件，默认输出到 `output/` 目录。
这份报告适合人工快速浏览，查看当天有哪些新岗位和哪些岗位更值得关注。

### 第一版代码的业务流程

一次完整运行的顺序如下：

1. 读取 `config.yaml`。
2. 创建必要的本地目录。
3. 从 `sources` 中逐个抓取页面。
4. 提取岗位标题、公司、地点、链接、发布时间等字段。
5. 对岗位进行清洗和去重。
6. 读取历史快照并比较差异。
7. 对新增岗位和变化岗位做匹配判断。
8. 生成 PDF 报告。
9. 保存新的历史快照。

### 第一版代码的输入

当前版本主要依赖这些输入：

- `config.yaml` 中的 `sources`
- `config.yaml` 中的 `keywords`
- `config.yaml` 中的 `companies`
- `config.yaml` 中的 `runtime` 配置
- `config.yaml` 中的 `model_backend` 配置

配置样例：

```yaml

  - https://example.com/careers
keywords:
  - Python
  - 数据分析
companies:
  - 示例科技
runtime:
  output_dir: output
  data_dir: data
  logs_dir: logs
  timeout_seconds: 10
model_backend:
  backend: rule
  api_base_url: ""
  api_key: ""
  model: gpt-4o-mini
  timeout_seconds: 30
  temperature: 0.2
  max_tokens: 512
```

### 第一版代码的输出

每次运行后，程序会生成这些结果：

- 终端输出运行摘要
- `output/` 下的 PDF 报告
- `data/history.json` 历史快照文件

## 第一版代码不能做什么

为了避免误解，下面这些能力当前不支持，或者只保留了接口，没有做完整实现：

### 1. 不能自动搜索全网招聘信息

当前版本只会抓取配置文件里明确写入的 URL。
它不会自动使用搜索引擎去发现新的招聘页面，也不会自动遍历全网公司站点。

### 2. 不能稳定处理复杂的动态网页

当前抓取逻辑使用 `requests` 请求页面 HTML，再用 `BeautifulSoup` 解析。
如果页面内容依赖 JavaScript 渲染、滚动加载、异步接口或登录态，第一版通常无法直接处理。

### 3. 不能绕过反爬和验证码

当前版本没有做代理池、验证码识别、浏览器自动化、指纹伪装等反爬处理。
如果目标站点限制请求，它可能会抓取失败。

### 4. 不能保证适配所有招聘网站结构

第一版是通用解析策略，不是每个站点的深度适配器。
如果页面结构特殊，可能需要后续补专门的站点解析器。

### 5. 不能做真正的 LLM 推理匹配

当前的 `LLMAnalyzer` 只是一个规则型代理，目的是先把业务流程跑通。
它会按关键词和公司名打分，但还没有接入真实的大模型 API。

### 6. 不能长期后台守护或自动定时运行

当前版本是一次性执行模型，不包含调度器、守护进程或定时任务框架。
如果要每天自动运行，需要后续再接 cron、系统任务或独立调度模块。

### 7. 不能做复杂的职位画像建模

第一版只支持简单的关键词和公司名匹配。
它不会分析完整的候选人画像、技能图谱或多层级岗位偏好。

## 第二版代码

这一版代码已经具备以下能力：

1. 读取本地 `config.yaml` 中的 `job_role`、`top_x`、`runtime` 和 `model_backend` 配置。
2. 先让模型根据岗位名称推断最可能发布该岗位招聘信息的前 x 家公司。
3. 自动抓取这些公司的招聘官网或招聘入口页面。
4. 提取页面中的文本、链接和日期线索，作为招聘信息分析依据。
5. 再把抓取结果交给第二次分析，重点识别最近一次招聘信息的发布时间。
6. 将整轮结果保存为本地 JSON 和 Markdown 文件，便于复查。

### 第二版代码能做什么

#### 1. 自动跑通“岗位 -> 公司 -> 官网 -> 分析”流程

运行 `main.py` 后，程序会按照配置里的岗位名称启动整个链路，先选公司，再抓官网，再做二次分析。
它不是后台常驻服务，而是“执行一次、输出一次结果”的批处理模式。

#### 2. 根据岗位名称筛选公司

第一阶段会把岗位名称写进提示词，交给模型或启发式后端处理。
输出结果会包含前 x 家最可能发布该岗位招聘信息的公司，以及它们的招聘官网入口。

#### 3. 抓取招聘官网页面

程序会依次访问这些公司招聘页，提取页面标题、正文文本、链接和时间线索。
如果目标页面里已经包含发布时间、招聘列表或相关说明，第二版通常可以识别出来。

#### 4. 做第二次招聘发布时间分析

抓取到的数据会再次交给模型或启发式分析器，重点判断最近一次招聘信息是什么时候发布的。
分析结果会返回公司名、发布时间、证据和简要总结。

#### 5. 保存运行结果

程序会把整轮执行结果写入本地文件：

- `output/job_watch_result.json`
- `output/job_watch_report.md`
- `output/job_watch_snapshot.json`

这让结果既能机器读取，也能人工快速浏览。

### 第二版代码的业务流程

一次完整运行的顺序如下：

1. 读取 `config.yaml`。
2. 创建必要的本地目录。
3. 读取岗位名称和候选公司数量配置。
4. 让模型先输出前 x 家最可能发布该岗位招聘信息的公司及招聘官网。
5. 依次访问这些官网招聘页面。
6. 提取页面中的文本、链接和日期线索。
7. 将抓取到的数据交给第二次分析。
8. 判断最近一次招聘信息的发布时间。
9. 保存 JSON 结果、Markdown 报告和快照文件。

### 第二版代码的输入

当前版本主要依赖这些输入：

- `config.yaml` 中的 `job_role`
- `config.yaml` 中的 `top_x`
- `config.yaml` 中的 `runtime` 配置
- `config.yaml` 中的 `model_backend` 配置

配置样例：

```yaml
job_role: 前端工程师
top_x: 3
model_backend:
  backend: heuristic
  api_base_url: ""
  api_key: ""
  model: gpt-4o-mini
  timeout_seconds: 30
  temperature: 0.0
  max_tokens: 1024
runtime:
  output_dir: output
  timeout_seconds: 15
  max_crawl_chars: 12000
  max_links_per_page: 20
```

### 第二版代码的输出

每次运行后，程序会生成这些结果：

- 终端输出运行摘要
- `output/job_watch_report.md` 报告文件
- `output/job_watch_result.json` 结果文件
- `output/job_watch_snapshot.json` 快照文件

## 第二版代码不能做什么

为了避免误解，下面这些能力当前不支持，或者只保留了兜底逻辑，没有做完整实现：

### 1. 不能保证覆盖所有招聘站

当前版本会优先抓取模型给出的官网入口，如果官网结构特殊或者页面打不开，结果可能不完整。

### 2. 不能稳定处理复杂的动态网页

当前抓取逻辑主要使用 `requests` 请求页面 HTML，再用 `BeautifulSoup` 解析。
如果页面内容依赖 JavaScript 渲染、滚动加载、异步接口或登录态，第二版通常无法直接处理。

### 3. 不能绕过反爬和验证码

当前版本没有做代理池、验证码识别、浏览器自动化、指纹伪装等反爬处理。
如果目标站点限制请求，它可能会抓取失败。

### 4. 不能保证发布时间一定能识别出来

如果页面里没有明显的日期文本、时间标签或可访问的招聘详情页，分析结果可能显示“未识别”。

### 5. 不能保证模型输出一定完全合规

如果你切换到 OpenAI 兼容后端，模型需要返回符合约定的 JSON。
如果模型输出格式不稳定，程序可能会退回到错误信息或不完整结果。

### 6. 不能长期后台守护或自动定时运行

当前版本是一次性执行流程，不包含调度器、守护进程或定时任务框架。
如果要每天自动运行，需要后续再接 cron、系统任务或独立调度模块。
